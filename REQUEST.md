# GAZO98 ～画嬢九八～

## PC-9801 エロゲー風画像コンバーター 実装仕様書

---

## 1. プロジェクト概要

### 1.1 コンセプト

現代のフルカラー画像を、PC-9801のエロゲーCG風（4096色中16色、ディザリング表現）に変換するWebアプリケーション。
98時代のドット絵職人が手作業で行っていた減色＋ディザリングを、アルゴリズムで再現する。

### 1.2 プロジェクト名

- 正式名称: **GAZO98**
- 副題: **画嬢九八**
- 読み: がぞう きゅうはち

### 1.3 技術スタック

- フロントエンド: HTML5 / CSS3 / Vanilla JavaScript（フレームワーク不使用）
- 画像処理: Canvas API + Web Worker
- サーバー: **不要**（クライアント完結）
- 対応ブラウザ: Chrome / Firefox / Safari / Edge（最新版）

## 2. 画像処理パイプライン

### 2.1 処理フロー

```
画像読み込み（File API / Drag & Drop）
  ↓
[Phase 1] クロップ（8:5固定比率、ドラッグ範囲選択）
  ↓
[Phase 2] リサイズ（ユーザー選択の出力解像度へ）
  ↓
[Phase 3] パレット生成（16色パレットの決定）
  ↓
[Phase 4] 減色＋ディザリング（選択手法で16色へ）
  ↓
[Phase 5] ポストエフェクト（CRT / スキャンライン等）
  ↓
プレビュー表示（スワイプ比較）
  ↓
ファイル出力（PNG / BMP / MAG）
```

### 2.2 Phase 1: クロップ

- 画像読み込み直後に**必ず**クロップUIを表示する
- クロップ枠のアスペクト比は **8:5 固定**（PC-9801の640×400に準拠）
- 操作:
  - ドラッグで選択範囲の位置を移動
  - ハンドルドラッグで選択範囲のサイズを変更（8:5比率を維持）
  - マウスホイールまたはピンチで拡大/縮小
- クロップ枠外は半透明の黒でオーバーレイ
- 「決定」ボタンで確定し、Phase 2へ進む

### 2.3 Phase 2: リサイズ

出力解像度を以下から選択（ドロップダウン）:

| 解像度    | 説明                         |
|-----------|------------------------------|
| 640×400   | PC-9801 標準（デフォルト）   |
| 640×200   | PC-9801 初期モード           |
| 320×200   | ハーフサイズ（軽量）         |
| 入力維持  | クロップ後のサイズをそのまま |

- リサイズアルゴリズム: バイリニア補間
- 8:5比率はクロップで確保済みのため、リサイズ時の歪みは発生しない
- 「入力維持」選択時は、クロップ後のピクセルサイズをそのまま使用

### 2.4 Phase 3: パレット生成

16色パレットの決定方式を以下の3つから選択:

#### A. 自動減色（デフォルト）

- **メディアンカット法**で画像の色空間を分析し、最適な16色を自動選出
- ただし、パレット値は **PC-9801のアナログパレット仕様** に準拠:
  - RGB各4bit（0〜15の16段階）
  - つまり 16×16×16 = **4096色** の中から16色を選ぶ
- 自動選出後、最も近い4096色グリッド値にスナップする

#### B. プリセットパレット

以下のプリセットを用意（代表的な98エロゲの色傾向を再現）:

| プリセット名       | 説明                                        |
|--------------------|---------------------------------------------|
| 標準（default）    | 汎用的な肌色・背景色バランス               |
| 暖色系（warm）     | エルフ系 - 肌色・茶系が豊富                |
| 寒色系（cool）     | SF・ホラー系 - 青・紫が豊富                |
| セピア（sepia）    | レトロ調 - 茶・黄系グラデーション          |
| 高コントラスト     | アリスソフト系 - 原色寄りでメリハリ重視    |
| モノクロ           | 白黒2値〜グレースケール16階調             |

- 各プリセットは16色の固定パレット値（RGB各4bit）を持つ
- プリセット選択後、手動編集に移行可能

#### C. 手動パレット編集

- 98風ダイアログでパレットエディタを開く
- 16色をそれぞれ個別に編集可能
- カラーピッカーは **4096色グリッド** に制限:
  - R: 0〜F（16段階スライダー）
  - G: 0〜F（16段階スライダー）
  - B: 0〜F（16段階スライダー）
- パレットのインポート/エクスポート機能（JSON形式）
- 「自動減色の結果を微調整」のユースケースを想定

### 2.5 Phase 4: ディザリング

以下の4種から選択（デフォルト: 市松パターン）:

#### 1. 市松パターン（Checkerboard）【デフォルト】

- 2×2の市松配置で2色を交互に配置
- PC-9801のエロゲCGで最も多用された手法
- 肌のグラデーション再現に最適
- 実装: 座標(x,y)の偶奇で2色を切り替え
- 閾値を段階的に変化させて多段階のグラデーションを表現

#### 2. 組織的ディザ（Ordered Dither / Bayer行列）

- 4×4 Bayer行列（デフォルト）、2×2 / 8×8 も選択可能
- 規則的なパターンでドット感が強い
- パラメータ: 行列サイズ（2×2 / 4×4 / 8×8）

#### 3. 誤差拡散（Floyd-Steinberg）

- 量子化誤差を周囲のピクセルに拡散
- 最も自然なグラデーション表現が可能
- 処理は重いが、Web Workerで対応
- 拡散係数:
  ```
        *   7/16
  3/16 5/16 1/16
  ```

#### 4. ランダムディザ

- 各ピクセルにランダムな閾値を適用
- ノイジーだが独特の質感
- シード値を固定可能（再現性のため）

#### 共通パラメータ

- **ディザ強度**: 0%〜100%スライダー（0%=ディザなしベタ塗り）
- 強度はディザパターンの閾値範囲に乗算で適用

### 2.6 Phase 5: ポストエフェクト

以下の3種のエフェクトを個別にON/OFF＋強度調整可能:

#### 1. スキャンラインエフェクト

- CRTディスプレイの走査線を再現
- パラメータ:
  - 強度: 0%〜100%（暗線の透明度）
  - 間隔: 1px / 2px / auto（出力解像度に応じて自動）
- 実装: 奇数ラインに対して黒を一定のアルファ値で合成

#### 2. CRTにじみ（RGB Shift + Glow）

- ブラウン管特有の色にじみを再現
- パラメータ:
  - RGBずれ量: 0〜3px（R/G/Bチャンネルを水平方向にずらす）
  - グロー強度: 0%〜100%（明るいピクセル周囲のぼかし）
  - グロー半径: 1〜5px
- 実装:
  - RGBずれ: 各チャンネルを個別にオフセットして合成
  - グロー: 明度閾値以上のピクセルにガウシアンブラーを適用し、加算合成

#### 3. ドット感強調（Sharpness）

- ピクセルの境界を強調してドット絵感を出す
- パラメータ:
  - 強度: 0%〜100%
- 実装: 最近傍補間でのスケーリング＋エッジ強調フィルタ
- 表示倍率が等倍以外の場合に特に効果的

#### エフェクト適用順序

```
ディザリング結果 → ドット感強調 → CRTにじみ → スキャンライン
```

## 3. ファイル入出力

### 3.1 入力

- 対応形式: JPEG / PNG / BMP / WebP / GIF（静止画のみ）
- 読み込み方法:
  - ファイル選択ダイアログ（98風UIの「ファイル」メニューから）
  - ドラッグ＆ドロップ（メイン画面へ）
- 最大ファイルサイズ: 制限なし（ブラウザのメモリに依存）

### 3.2 出力形式

#### PNG

- 出力解像度のまま保存
- インデックスカラー（16色パレット）で出力
- ポストエフェクト適用後はフルカラーPNGとなる

#### BMP

- Windows BMP形式
- 4bit（16色）インデックスカラー
- ポストエフェクト適用版はBMP非対応（PNG推奨）とする

#### MAG形式

- MAKI-chan Graphic format（まきちゃんグラフィックローダー用）
- PC-9801実機で表示可能なフォーマット
- 仕様:
  - ヘッダ: "MAKI02  " (8byte) + コメント + 0x1A
  - 機種ID: PC-9801（98h）
  - ヘッダフラグ: 200ラインモード/400ラインモード
  - パレットデータ: 16色 × GRB各4bit
  - 画像データ: フラグ＋ピクセルのランレングス圧縮
  - ピクセルはXOR差分＋フラグベースの圧縮方式
- エンコード手順:
  1. 16色パレットをGRB順に格納
  2. 横方向のピクセルペア（2ピクセル=1byte）を生成
  3. 前行とのXOR差分を計算
  4. フラグA（行単位の繰り返し判定）生成
  5. フラグB（ピクセルペア単位の一致判定）生成
  6. フラグとピクセルデータを交互に格納
- MAG出力時はポストエフェクトなしの純粋な16色データを使用

### 3.3 出力オプション

- ファイル名: デフォルトは `gazo98_YYYYMMDD_HHMMSS.{ext}`
- エフェクトなし版も同時保存するオプション
- MAG出力時はポストエフェクト非適用を自動選択

## 5. Web Worker設計

### 5.1 Worker分離対象

以下の重い処理をWeb Workerに委譲:

- メディアンカット減色アルゴリズム
- Floyd-Steinberg誤差拡散処理
- MAG形式エンコード
- CRTグローエフェクト（ガウシアンブラー）

### 5.2 メッセージプロトコル

```
メインスレッド → Worker:
{
  type: "convert",
  imageData: ImageData,      // Canvas ImageData
  options: {
    palette: [...],          // 16色パレット (or "auto")
    dither: "checkerboard",  // ディザリング手法
    ditherStrength: 80,      // 強度 0-100
    resolution: [640, 400],  // 出力解像度
    bayerSize: 4             // Bayer行列サイズ（ordered時）
  }
}

Worker → メインスレッド:
{
  type: "progress",
  phase: "dithering",
  percent: 45
}

{
  type: "result",
  imageData: ImageData,      // 変換後ImageData
  palette: [...]             // 使用パレット
}
```

### 5.3 プログレス表示

- Worker処理中は98風プログレスバーを表示:
  ```
  ┌────────────────────────┐
  │  変換中...              │
  │  [████████░░░░░░] 53%  │
  │                        │
  │       [中止]           │
  └────────────────────────┘
  ```
- 中止ボタンでWorkerをterminate可能

## 7. MAG形式 技術仕様

### 7.1 ファイル構造

```
[ヘッダ部]
  +0x00: "MAKI02  " (8 bytes, ASCII)
  +0x08: 機種ID + ユーザー名 (可変長, コメント)
  +0x?? : 0x1A (1 byte, コメント終端)

[ヘッダ情報] (offset = コメント終端+1 からの相対)
  +0x00: top (1 byte) - 先頭Y座標上位
  +0x01: 機種フラグ (1 byte)
          bit7: 0=200line, 1=400line
          bit0-1: 機種ID (0=PC-9801)
  +0x02: 左端X (2 bytes, little-endian)
  +0x04: 上端Y (2 bytes, little-endian)
  +0x06: 右端X (2 bytes, little-endian)
  +0x08: 下端Y (2 bytes, little-endian)
  +0x0A: フラグAオフセット (4 bytes, little-endian)
  +0x0E: フラグBオフセット (4 bytes, little-endian)
  +0x12: フラグBサイズ (4 bytes, little-endian)
  +0x16: ピクセルオフセット (4 bytes, little-endian)
  +0x1A: ピクセルサイズ (4 bytes, little-endian)

[パレット] (32 bytes)
  16色 × 2 bytes (GRB: 上位4bit=G, 中位4bit=R, 下位4bit=B)

[フラグAデータ]
[フラグBデータ]
[ピクセルデータ]
```

### 7.2 圧縮アルゴリズム概要

1. 画像を2ピクセル単位（1byte = 上位4bit:左 + 下位4bit:右）に変換
2. 各行について前の行とのXOR差分を取る
3. XOR結果が0の連続（＝前行と同一）をフラグAで管理
4. フラグBでピクセルペア単位の一致パターンを管理
5. フラグに合致しないデータのみピクセルデータとして格納

## 9. 非機能要件

### 9.1 パフォーマンス目標

- 640×400画像の変換: 2秒以内（市松/Bayer）、5秒以内（Floyd-Steinberg）
- プレビュー更新: パラメータ変更後500ms以内にデバウンス実行
- Web Worker使用により、UI操作は処理中もブロッキングしない

### 9.2 ブラウザ互換性

- Chrome 90+ / Firefox 90+ / Safari 15+ / Edge 90+
- Web Worker, Canvas 2D, File API, Blob, OffscreenCanvas対応が必須

### 9.3 セキュリティ

- すべての処理はクライアントサイドで完結
- 画像データはサーバーに送信しない
- 外部リソースへのアクセスはWebフォントのみ
